# Lumina Architecture

Last reviewed: 2026-06-30

## Purpose

Lumina is a personal finance PWA focused on daily cashflow: users register incomes and expenses, then the app shows the monthly balance, recent movements, categories, recurring commitments, and proactive alerts.

The production build runs as a static Vite app on Vercel and uses Supabase for authentication, database persistence, row-level security, views, and alert triggers.

## Runtime Shape

- Frontend: React, Vite, Tailwind CSS, lucide-react.
- Backend services: Supabase Auth and Supabase Postgres.
- Hosting: Vercel, connected to the GitHub repository.
- PWA layer: `public/manifest.webmanifest`, `public/sw.js`, `public/offline.html`, and generated icons.
- Local fallback: when Supabase environment variables are missing, Lumina keeps a demo mode alive with local seed data.

## Main Application Flow

1. The app starts in `src/App.jsx`.
2. `src/lib/supabase.js` creates the Supabase client only when both Vite environment variables exist.
3. `src/services/auth.js` handles sign in, sign up, sign out, current session, and auth state changes.
4. After a valid session, the app loads user-owned rows through service modules:
   - `src/services/transactions.js`
   - `src/services/incomes.js`
   - `src/services/recurring.js`
   - `src/services/alerts.js`
5. New users are provisioned by the database trigger `public.handle_new_user()`, which creates a profile and default categories.

## Data Model

Core tables:

- `profiles`: one row per authenticated user, with currency and legacy budget metadata.
- `categories`: user-scoped spending categories.
- `transactions`: expense records.
- `incomes`: income records.
- `recurring_expenses`: expected recurring payments.
- `alerts`: proactive notices generated by database triggers or shown by the client fallback.

Analytic views:

- `monthly_spending`: monthly expense total per user.
- `monthly_income`: monthly income total per user.
- `monthly_cashflow`: monthly income minus monthly spending.
- `monthly_recurring_commitments`: active recurring commitments per user.

## Security Boundary

The browser receives only the Supabase anonymous key through Vite environment variables:

- `VITE_SUPABASE_URL`
- `VITE_SUPABASE_ANON_KEY`

User data isolation is enforced in Supabase with Row Level Security. The public tables use policies based on `auth.uid()`, so a logged-in user can only read and write rows tied to their own user id.

No service-role key should ever be used in the frontend or in Vercel environment variables for this app.

## Deployment Flow

1. Changes are committed to `main`.
2. GitHub receives the push.
3. Vercel builds the static app with `npm run build`.
4. Vercel publishes `dist`.
5. Supabase credentials are injected at build time through Vercel Environment Variables.

When Supabase schema changes are needed, run the SQL in Supabase first, then deploy the frontend that depends on it.

## Operational Notes

- The Supabase email confirmation flow must be configured to return users to `https://lumina-gastos.vercel.app`.
- For every new database capability, prefer a migration file under `supabase/migrations/` rather than editing only the initial schema.
- The demo mode is for development and public preview without credentials; authenticated production users should start from empty user-owned data.
- `tools/superpowers` is a local audit/reference checkout and is intentionally ignored by Git.
